The technical decisions, the failures, and the lessons learned while building a hardware-aware AI product.

Building NoWires.AI in Public

We’ve decided to build NoWires.AI transparently. Here’s the first deep dive.

The canvas question

The biggest early decision: how to render the breadboard?

Options we considered:

  1. SVG — clean, accessible, easy to animate. Problem: gets slow with 830+ nodes and live wire updates.
  2. React (DOM elements) — familiar, but React’s reconciliation overhead would hurt interactivity.
  3. HTML5 Canvas 2D — fast, imperative, no framework overhead.

We went with Canvas 2D + a pure TypeScript renderer class.

The key insight: keep React out of the render loop entirely. The renderer is a standalone class that takes pure state and renders it. React owns the canvas element; the renderer owns what’s inside it. This means tests can run the renderer without a DOM, and the render loop is always exactly as fast as the math behind it.

export class BreadboardRenderer {
  constructor(ctx: CanvasRenderingContext2D) { ... }

  renderAll(state: RenderState): void {
    // Pure function of state → pixels
    // No React, no lifecycle, no diffing
  }
}

The netlist challenge

Computing which holes are electrically connected sounds simple. It isn’t.

A breadboard has implicit connectivity (5-hole strips) and explicit connectivity (wires). Building a netlist means union-finding all connected holes across both types.

We used a union-find (disjoint sets) algorithm, seeding it with both implicit strip connections and explicit wire connections. The result: a Map<holeKey, netId> that updates every time you add or remove a wire.

What we got wrong

First mistake: we tried to store canvas render state inside Zustand. Every mousemove was triggering React re-renders. The fix was obvious in hindsight — keep hover state in a ref, not in the store.

Second mistake: the hitTest function was O(n) over all 830 holes on every mousemove. We added a simple spatial hash and it’s now O(1).

Third mistake: we forgot that TypeScript’s exactOptionalPropertyTypes flag breaks a lot of common patterns. Worth knowing about upfront.
